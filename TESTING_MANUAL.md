# Blabber — Руководство по ручному тестированию

> **Как пользоваться этим файлом:**
> Каждый пункт — конкретное действие в Telegram с ожидаемым результатом.
> После успешной проверки ставь `[x]` вместо `[ ]`.
> Если что-то пошло не так — пиши заметку под пунктом.

---

## Подготовка к тестированию

### Нужно знать перед стартом

1. **Узнай свой Telegram ID** — напиши [@userinfobot](https://t.me/userinfobot), он ответит числом вида `123456789`.

2. **Убедись, что `.env` настроен.** Найди и раскомментируй строку:
   ```
   ADMIN_TELEGRAM_IDS=ТВОЙ_ID_ЗДЕСЬ
   ```

3. **Перезапусти бота** после изменения `.env`:
   ```bash
   sudo systemctl restart blabber
   sudo systemctl status blabber
   ```
   Статус должен быть `Active: active (running)`.

4. **Для тестов с двумя пользователями** — нужен второй Telegram-аккаунт. Попроси его написать боту `/start` и узнай его Telegram ID.

### Как смотреть логи при ошибке
```bash
sudo journalctl -u blabber -n 50 --no-pager
```

## Если что-то сломалось

### Смотреть логи
```bash
# Последние 50 строк
sudo journalctl -u blabber -n 50 --no-pager

# В реальном времени
sudo journalctl -u blabber -f
```

### Быстрый рестарт
```bash
sudo systemctl restart blabber && sudo systemctl status blabber
```

### Сбросить БД полностью (ядерный вариант)
```bash
sudo systemctl stop blabber
rm blabber.db
sudo systemctl start blabber
# БД пересоздастся с нуля при следующем запуске
```

### Проверить что `.env` загружен правильно
```bash
cd /home/ezovskikh_a/apps/blabber
grep ADMIN_TELEGRAM_IDS .env
grep DATABASE_PATH .env
```

### Проблемы с контекстом (режим Чат)

| Симптом | Возможная причина |
|--------|-------------------|
| Бот не помнит предыдущие реплики | Режим "Вопрос-ответ" — переключи `/mode chat` |
| История очистилась сама | TTL 60 минут — долго не писал, контекст сбросился |
| Кнопки `/reset` не работают | Callback привязан к user_id — другой пользователь не может нажать |
| Ошибка при `/mode` | Проверь миграцию 004: `context_mode` в `users`, таблицы `context_messages`, `context_summary` |

```bash
sqlite3 blabber.db ".schema context_messages"
sqlite3 blabber.db ".schema context_summary"
sqlite3 blabber.db "PRAGMA table_info(users);" | grep context_mode
```

### Проблемы с памятью (профиль и KB)

| Симптом | Возможная причина |
|--------|-------------------|
| `/remember` или `/prefer` не работает | Проверь миграцию 005: `user_profiles` (+ миграцию 009: `user_profiles.kind`) |
| `/memory` не работает | Проверь миграцию 009: `memory_suggestions`, поля `users.auto_memory_*` |
| `/quote` не работает | Проверь миграцию 010: таблица `quotes`; проверь `lancedb` в venv |
| `/quotes search` не ищет по смыслу | `OPENAI_API_KEY` не задан — работает keyword-поиск (это нормально) |
| Ошибка `ModuleNotFoundError: lancedb` | Запусти `venv/bin/pip install lancedb pyarrow` |
| `/kb` не работает | Проверь миграцию 006: `kb_documents`, `kb_chunks` |
| Embeddings = NULL при загрузке | `OPENAI_API_KEY` не задан или невалиден — fallback на BM25-only |
| «BM25-only, no API key» при загрузке | Нормально если `OPENAI_API_KEY` не настроен; качество поиска ниже |
| Ошибка при загрузке PDF | Не установлен `pypdf`: `pip install pypdf` |
| Ошибка при загрузке DOCX | Не установлен `python-docx`: `pip install python-docx` |

```bash
sqlite3 blabber.db ".schema user_profiles"
sqlite3 blabber.db ".schema kb_chunks"
sqlite3 blabber.db "PRAGMA table_info(kb_chunks);" | grep embedding
```

## Раздел 15c — KB Migration Safety Checks

> **Что проверяем:** безопасную миграцию KB с legacy `embedding BLOB` на `LanceDB` без регрессий в пользовательском сценарии.
>
> **Статус:** активный раздел. Актуален после завершения Sprint 1–6.

- [ ] **15c.1** Legacy path остается рабочим после включения `dual-write`
  - Загрузить документ в KB
  - ✅ Документ индексируется и доступен для ответов
  - ✅ При недоступном `LanceDB` пользовательский сценарий не ломается

- [ ] **15c.2** Новый документ пишет vector entries в `LanceDB`
  - Загрузить документ при включенном `KB_ENABLE_DUAL_WRITE`
  - ✅ В `SQLite` есть документ и чанки
  - ✅ В `LanceDB` появились соответствующие vector entries

- [ ] **15c.3** Ошибка `LanceDB` не ломает загрузку документа
  - Искусственно сломать доступ к `LanceDB`
  - ✅ Бот не падает
  - ✅ Документ остается доступным хотя бы через legacy path / `BM25-only`

- [ ] **15c.4** Read path можно вернуть на legacy backend
  - Переключить `KB_VECTOR_BACKEND=sqlite`
  - ✅ Retrieval снова идет по старому пути
  - ✅ `/kb` и ответы по документам продолжают работать

- [ ] **15c.5** `LanceDB`-retrieval не ухудшает ответы по KB
  - Включить новый backend
  - Задать вопросы по документам с точными совпадениями и синонимами
  - ✅ Ответы корректны и не хуже legacy path

- [ ] **15c.6** Graceful degradation без embeddings
  - Убрать `OPENAI_API_KEY`
  - ✅ KB продолжает отвечать через `BM25-only`
  - ✅ Нет падений индексации и retrieval

- [ ] **15c.7** Удаление документа чистит оба слоя хранения
  - Удалить документ через `/kb`
  - ✅ Чанки удалены из `SQLite`
  - ✅ Vector entries удалены из `LanceDB`

- [ ] **15c.8** Повторная индексация не создает неконтролируемых дублей
  - Проиндексировать один и тот же источник повторно
  - ✅ Поведение предсказуемо и задокументировано
  - ✅ Нет необъяснимого роста дублей в `LanceDB`

- [ ] **15c.9** URL ingestion включён и работает в общей KB
  - Проверить флаг `KB_ENABLE_URL_INGESTION`
  - ✅ `/kb url https://...` доступен пользователю
  - ✅ URL-страницы попадают в тот же KB pipeline, что и файлы

- [ ] **15c.10** Legacy BLOB по умолчанию больше не пополняется новыми документами
  - Убедиться, что `KB_WRITE_LEGACY_EMBEDDING=false`
  - Загрузить новый документ
  - ✅ KB работает через `LanceDB`
  - ✅ При необходимости rollback buffer можно освежить отдельно через `KB_WRITE_LEGACY_EMBEDDING=true` + `/kb reindex all`

- [ ] **15c.11** Agent может сохранить URL в KB
  - Выполнить `/agent kb https://example.com/`
  - ✅ Страница сохраняется в ту же KB
  - ✅ После этого `/kb` показывает URL-источник
